## Tools Used and Why

### Docker Desktop

**What it is:** Docker Desktop is an application that runs containerised software on your local machine. A *container* is a lightweight, isolated environment that packages an application and all its dependencies together, so it behaves identically regardless of what's installed on your OS.

**Why we use it here:** PostgreSQL has a notoriously complex Windows installation — you have to manage service accounts, `pg_hba.conf`, PATH entries, and version conflicts. Docker lets you run the exact same PostgreSQL 16 image that would run in production with a single command (`docker compose up -d`). When you're done, `docker compose down` removes it cleanly. No leftover Windows services, no version conflicts.

**Why WSL2 backend:** Docker containers are Linux processes. On Windows they need a Linux kernel to run on. WSL2 (Windows Subsystem for Linux 2) provides a real Linux kernel inside Windows — this is faster and more compatible than the older Hyper-V backend. Windows 11 Home supports WSL2 but not Hyper-V, so WSL2 is the only option. Docker Desktop installs WSL2 automatically.

**Why Docker Compose specifically:** Rather than running raw `docker run` commands with long flag lists, Docker Compose lets you describe your services in a `docker-compose.yml` file. This is version-controlled, reproducible, and easy to extend (we'll add Qdrant in Phase 5 just by adding another service block).

---

### SQLAlchemy

**What it is:** SQLAlchemy is a Python library that maps Python classes (models) to database tables (ORM = Object-Relational Mapper). Instead of writing raw SQL strings throughout the app, you work with Python objects and SQLAlchemy generates the SQL.

**Why we use it:** FastAPI is async-first and SQLAlchemy 2.0 has native async support via `asyncpg`. This means database queries don't block the server while waiting for a result — the event loop can serve other requests. We use `sqlalchemy[asyncio]` for the app and `psycopg2-binary` only for Alembic (which runs as a CLI tool outside the async loop).

---

### Alembic

**What it is:** Alembic is a database migration tool for SQLAlchemy. A *migration* is a versioned, tracked change to the database schema — like adding a column, creating a table, or dropping an index. Each migration is a Python file in `alembic/versions/`.

**Why we use it instead of `Base.metadata.create_all()`:** `create_all()` creates tables but can never alter them — if you add a column later, it silently does nothing on an existing database. Alembic tracks which migrations have run and applies only the new ones. This is the standard production pattern.

**The sync/async split:** Alembic can't use an async engine directly (it's a CLI tool, not a server). So it uses `psycopg2-binary` (sync driver) while the actual FastAPI app uses `asyncpg` (async driver). Both connect to the same database — the driver choice only affects how Python makes the TCP connection.

---

### asyncpg

**What it is:** asyncpg is a pure-Python async driver for PostgreSQL. It implements the PostgreSQL wire protocol natively using Python's `asyncio`, with no C extension wrapping required.

**Why asyncpg over psycopg2:** psycopg2 is synchronous — it blocks the current thread while waiting for PostgreSQL to respond. In an async web server, blocking a thread means no other requests can be served during that wait. asyncpg is non-blocking: while waiting for the DB, the event loop handles other incoming requests. This matters most under concurrent load.

---

### pydantic-settings

**What it is:** An extension of Pydantic that reads configuration from environment variables (and `.env` files) and validates them with Pydantic's type system.

**Why we use it:** All secrets (DB passwords, API keys) and environment-specific config (dev vs prod URLs) live in `.env` which is gitignored. `pydantic-settings` reads them, validates types, and exposes them as a typed `Settings` object. The `@lru_cache` wrapper means the settings are loaded once and reused — no repeated disk reads.

## Database Table Schemas

All five tables are in the `chiefofstaff` PostgreSQL database. The schema is student-facing: a single student's academic life — what they're enrolled in, what assignments exist, and how they're doing.

### `students`

The root entity. One row per student (currently one: Alex Johnson).

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY | Auto-incrementing surrogate key |
| `name` | VARCHAR(100) | NOT NULL | Student's full name |
| `email` | VARCHAR(200) | NOT NULL, UNIQUE | University email address |
| `major` | VARCHAR(100) | NOT NULL | Declared major (e.g. "Computer Science") |
| `year_of_study` | INTEGER | NOT NULL | Academic year (1 = freshman, 4 = senior) |

---

### `courses`

One row per course offered. Independent of any student — courses exist regardless of who's enrolled.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY | Auto-incrementing surrogate key |
| `code` | VARCHAR(20) | NOT NULL, UNIQUE | Course code, e.g. "CS201" |
| `name` | VARCHAR(200) | NOT NULL | Full course name |
| `credits` | INTEGER | NOT NULL | Credit hours (2–4) |
| `semester` | VARCHAR(20) | NOT NULL | Term identifier, e.g. "2025-Spring" |
| `teacher` | VARCHAR(100) | NOT NULL | Instructor name |

---

### `enrollments`

The join table between students and courses. Also stores per-student, per-course performance data.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY | Auto-incrementing surrogate key |
| `student_id` | INTEGER | NOT NULL, FK → students.id | Which student |
| `course_id` | INTEGER | NOT NULL, FK → courses.id | Which course |
| `current_grade` | FLOAT | NULLABLE | Running average score across submitted work |
| `attendance_pct` | FLOAT | NOT NULL, DEFAULT 100.0 | Percentage of classes attended |

**Unique constraint:** `(student_id, course_id)` — a student can only be enrolled in a course once.

---

### `assignments`

One row per assignment. Belongs to a course, not a student — the assignment exists regardless of who submits it.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY | Auto-incrementing surrogate key |
| `course_id` | INTEGER | NOT NULL, FK → courses.id | Which course this belongs to |
| `title` | VARCHAR(200) | NOT NULL | Assignment name |
| `type` | VARCHAR(50) | NOT NULL | One of: `homework`, `quiz`, `midterm`, `final`, `project` |
| `due_date` | DATETIME | NOT NULL | Deadline (past = already happened, future = upcoming) |
| `max_score` | FLOAT | NOT NULL, DEFAULT 100.0 | Maximum possible score |
| `description` | TEXT | NULLABLE | Assignment description / instructions |

---

### `submissions`

One row per student-assignment pair. A row with `score = NULL` and `submitted_at = NULL` means the assignment was created (in seed data) but not yet submitted — these represent upcoming or not-yet-attempted work.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY | Auto-incrementing surrogate key |
| `student_id` | INTEGER | NOT NULL, FK → students.id | Which student |
| `assignment_id` | INTEGER | NOT NULL, FK → assignments.id | Which assignment |
| `score` | FLOAT | NULLABLE | Score received (NULL = not yet graded or not submitted) |
| `submitted_at` | DATETIME | NULLABLE | When it was submitted (NULL = not yet submitted) |
| `feedback` | TEXT | NULLABLE | Instructor feedback text |

**Unique constraint:** `(student_id, assignment_id)` — one submission per assignment per student.

## Step-by-Step: How Everything Was Set Up

This section documents the exact sequence of steps taken to go from a blank directory to a running stack.

### Step 1 — Create project scaffold

Created the directory tree (`backend/`, `frontend/`, `docs/`, subdirectories) with `__init__.py` files in every Python package and `.gitkeep` placeholders in empty directories so git tracks them. Also created `.gitignore` (ignoring `.env`, `.venv/`, `__pycache__/`, `*.py[cod]`, `node_modules/`, `.next/`, `*.log`).

---

### Step 2 — Install Docker Desktop

Docker Desktop is required to run PostgreSQL as a container.

1. Downloaded Docker Desktop installer from [docker.com](https://www.docker.com/products/docker-desktop/) — ~600 MB `.exe`.
2. Ran the installer with `--backend=wsl-2` flag for the WSL2 backend (required on Windows 11 Home — Hyper-V is not available on Home edition).
3. Installer automatically installed WSL2 and the Ubuntu distro it needs.
4. After install, refreshed the PowerShell `PATH` by reading it from the Machine and User environment variables — new installations don't update the running shell session automatically.
5. Opened Docker Desktop and waited for the engine to fully start (polled `docker info` until it returned exit code 0).

**Why Docker instead of a native PostgreSQL install:**
- Native installs leave Windows Services running in the background forever.
- Version management is painful — upgrading PostgreSQL on Windows requires a dump/restore.
- Docker containers are isolated: start them when you need them, stop them when you don't.
- `docker-compose.yml` is version-controlled, so any collaborator can reproduce the exact same database setup with one command.
- Docker is what production deployments use — using it locally removes an entire class of "works on my machine" issues.

---

### Step 3 — Write `docker-compose.yml`

Created at the project root. Defines a single service `postgres` using the official `postgres:16` image, with:
- Environment variables for the username, password, and database name (all `chiefofstaff` for local dev).
- Port mapping `5432:5432` so the app can connect on `localhost:5432`.
- A named Docker volume `postgres_data` that persists data between container restarts.
- A healthcheck that runs `pg_isready` every 5 seconds so `docker compose up` doesn't report ready until PostgreSQL is actually accepting connections.

---

### Step 4 — Create Python 3.10 virtual environment

The codebase uses `X | None` syntax for union types in SQLAlchemy `Mapped` columns (e.g. `Mapped[float | None]`). This syntax was introduced in Python 3.10. The system default Python was 3.9.13 (MSYS2).

```powershell
py -3.10 -m venv .venv
.\.venv\Scripts\Activate.ps1
```

The `py` launcher is a Windows tool that lets you choose which Python version to use by flag rather than by PATH order.

---

### Step 5 — Install Python dependencies

Created `requirements.txt` with all dependencies, then:

```powershell
pip install -r requirements.txt
```

Key packages and their roles:

| Package | Role |
|---|---|
| `fastapi` | Web framework — defines routes, request/response validation |
| `uvicorn[standard]` | ASGI server that actually runs FastAPI |
| `pydantic-settings` | Reads `.env` files into typed `Settings` objects |
| `sqlalchemy[asyncio]` | ORM + async DB session management |
| `asyncpg` | Async PostgreSQL driver (used by SQLAlchemy at runtime) |
| `alembic` | Database migration tool |
| `psycopg2-binary` | Sync PostgreSQL driver (used only by Alembic CLI) |
| `python-dotenv` | Loads `.env` into `os.environ` |
| `httpx` | Async HTTP client (used by health check to ping Ollama) |
| `langchain-ollama` | LangChain integration for Ollama models |
| `langchain-core` | LangChain base classes and interfaces |
| `langgraph` | Graph orchestration (used from Phase 4 onward) |

Resolution: changed both to `>=` and let pip resolve compatible versions. Production code would use `pip-compile` to generate a locked `requirements.txt` with exact compatible versions.

---

### Step 6 — Write config, DB base, and models

1. **`backend/config.py`** — `pydantic-settings` `Settings` class. Reads `DATABASE_URL`, `DATABASE_URL_SYNC`, `OLLAMA_BASE_URL`, etc. from `.env`. Uses `@lru_cache` so it's loaded once.

2. **`backend/db/base.py`** — creates the async SQLAlchemy engine using `asyncpg`, the `async_sessionmaker`, and the `Base` declarative base class all models inherit from. Also defines `get_db()`, a FastAPI dependency that yields a session and auto-commits or rolls back.

3. **`backend/db/models.py`** — five ORM model classes (`Student`, `Course`, `Enrollment`, `Assignment`, `Submission`) using SQLAlchemy 2.0's `Mapped` + `mapped_column` style (strongly typed, no `Column()` old-style).

---

### Step 7 — Set up Alembic

Alembic manages schema versions so you can evolve the database over time without dropping and recreating it.

```powershell
# From backend/
alembic init alembic
```

This created `alembic.ini` and `alembic/env.py`.

**Configuration changes made:**

- `alembic.ini`: Set `sqlalchemy.url` to the sync psycopg2 connection string (hardcoded — see the interpolation bug below).
- `alembic/env.py`: Imported `db.base.Base` and all models so Alembic can detect the schema. Set `target_metadata = Base.metadata`. Overrides the URL from `os.environ["DATABASE_URL_SYNC"]` at runtime.

**Bug encountered — ConfigParser interpolation:** Setting `sqlalchemy.url = %(DATABASE_URL_SYNC)s` in `alembic.ini` was intended to read an environment variable. Instead, Python's `ConfigParser` (which Alembic uses to parse `.ini` files) interpreted `%(...)s` as a ConfigParser variable reference — not an environment variable — and raised `InterpolationMissingOptionError`. Fixed by hardcoding the local dev URL directly in `alembic.ini` and letting `env.py` override it via `os.environ` in CI/production.

---

### Step 8 — Generate and apply the initial migration

```powershell
# From backend/ with venv active
alembic revision --autogenerate -m "initial schema"
alembic upgrade head
```

`--autogenerate` inspects `Base.metadata` (all the models) and generates a migration file in `alembic/versions/` that creates all five tables with the correct columns, types, and constraints.

`upgrade head` runs all pending migrations — in this case, the one we just generated.

**Port conflict encountered:** The local `postgresql-x64-16` Windows service was already running on port 5432. Docker couldn't bind to the port. Fixed by stopping the Windows service with elevated PowerShell:

```powershell
Start-Process powershell -Verb RunAs -ArgumentList "-Command Stop-Service postgresql-x64-16"
```

After stopping it, `docker compose up -d` started successfully and the migration applied cleanly.

---

### Step 9 — Seed the database

```powershell
# From backend/ with venv active
python -m db.seed
```

The seed script (`backend/db/seed.py`) inserts Alex Johnson, 5 courses, 5 enrollments, 23 assignments, and 23 submissions (with realistic past scores and NULL scores for future assignments). It's idempotent-safe to run once — running it twice will error on the unique constraints, which is intentional (prevents accidental double-seeding).

---

### Step 10 — Write FastAPI app and health endpoint

1. **`backend/schemas/`** — Pydantic response models: `HealthResponse`, `ErrorResponse`, `ChatRequest`, `ChatResponse`, `Message`, `MessageRole`.

2. **`backend/api/health.py`** — `GET /health` router. Runs `SELECT 1` against the DB and `GET /api/tags` against Ollama (with a 3-second timeout each). Returns the status of both separately so you can tell which dependency is down.

3. **`backend/main.py`** — FastAPI app with:
   - `asynccontextmanager` lifespan for startup/shutdown logging.
   - CORS middleware allowing `http://localhost:3000` (Next.js frontend).
   - Per-request UUID middleware — generates a UUID for every request and adds it as `X-Request-ID` to the response header. Used for correlating logs.
   - Global exception handler — catches any unhandled exception and returns a clean `500` JSON response instead of a raw 500.

## Key Decisions

### Async SQLAlchemy from the start

**Decision:** Use `sqlalchemy[asyncio]` with `asyncpg` for the app, and `psycopg2-binary` only for Alembic migrations.

**Rationale:** Phase 3 requires streaming responses and the chat endpoint will do async DB writes mid-request. Starting async avoids a painful rewrite later. Alembic uses a sync connection because it's a CLI tool run outside the async event loop — this is the standard pattern.

**Tradeoff:** Slightly more boilerplate (async sessions, `await` everywhere in DB code), but the right foundation for a production-style app.

---

### Python 3.10 virtual environment

**Decision:** Use Python 3.10.9 (via `py -3.10`) rather than the system default (3.9.13).

**Rationale:** The codebase uses the `X | None` union type syntax (e.g. `float | None` in SQLAlchemy `Mapped` columns), which requires Python 3.10+. Adding `from __future__ import annotations` to every file to work around this on 3.9 would have been fragile and non-obvious to future readers.

**Tradeoff:** Requires `py -3.10` to be available; Python 3.10.9 was already installed on this machine.

---

### Single student, realistic seed data

**Decision:** Seed a fixed single student (Alex Johnson) rather than a multi-user schema.

**Rationale:** The AI Chief of Staff is a personal assistant — there's no login or multi-tenancy needed. One student's data with varied scores, upcoming deadlines, and partial submissions makes the SQL agent demos interesting without extra complexity.

**Tradeoff:** Can't demonstrate multi-user isolation, but that's an explicit out-of-scope decision.

---

### Health endpoint checks both DB and Ollama

**Decision:** `GET /health` actively pings the DB with `SELECT 1` and Ollama with `GET /api/tags` and reports each separately.

**Rationale:** A health check that only returns `{"status": "ok"}` unconditionally is useless. Separating `db` and `ollama` fields lets you quickly identify which dependency is down.

**Tradeoff:** Adds ~100ms to the health check latency. Acceptable — health checks aren't on the critical path.

## Observations & Limitations

- **Port conflict with local PostgreSQL** — a local `postgresql-x64-16` Windows service was running on port 5432, blocking the Docker container. Had to stop it with elevated PowerShell. Going forward, the local service should remain stopped (or changed to a different port) when working on this project.

- **`alembic.ini` ConfigParser interpolation** — using `%(DATABASE_URL_SYNC)s` in `alembic.ini` triggers Python ConfigParser variable interpolation, not environment variable substitution. Fixed by hardcoding the URL in `alembic.ini` and having `env.py` override from `os.environ` at runtime.

- **LangChain strict version pinning conflict** — `langchain-ollama==0.2.3` and `langchain-core==0.3.25` conflicted when pinned exactly. Relaxed to `>=` constraints and let pip resolve. Production would use a `pip-compile`-generated lockfile instead.

- **Docker Desktop on Windows 11 Home** — requires the WSL2 backend (Hyper-V is not available on Home edition). Docker Desktop installs WSL2 automatically during setup, so no manual WSL configuration was needed.

- **SQL echo in development** — `echo=settings.app_env == "development"` in `create_async_engine` prints every SQL statement. Useful for debugging now, should be turned off before any performance testing.
